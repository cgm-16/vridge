## Findings

- Candidate 1: `pnpm lint` fails in a repo that uses nested git worktrees under `.worktrees/`
  - Reproduction:
    1. Keep at least one nested worktree with generated output, for example `.worktrees/vridge-T04/.next/build/...`
    2. Run `pnpm lint` from repo root
  - Evidence:
    - ESLint reports errors from `.worktrees/vridge-T04/.next/build/chunks/...`
    - `eslint.config.mjs` ignores `.next/**` only at repo root and does not ignore `.worktrees/**`
  - Impact:
    - Standard lint command fails for a normal multi-worktree checkout even when the current checkout source is clean
  - GitHub issue:
    - #39 `pnpm lint가 .worktrees 내부 Next.js 산출물까지 스캔해 실패함`
    - https://github.com/cgm-16/vridge/issues/39

- Candidate 2: `pnpm test` runs sibling-worktree suites and fails in a repo that uses nested git worktrees under `.worktrees/`
  - Reproduction:
    1. Keep at least one nested worktree under `.worktrees/`, for example `.worktrees/feat-migrate-helm-hook`
    2. Run `pnpm test` from repo root
  - Evidence:
    - Jest prints `Haste module naming collision: migration` between `tools/migration/package.json` and `.worktrees/feat-migrate-helm-hook/tools/migration/package.json`
    - Jest then executes suites from `.worktrees/*/__tests__`
    - `pnpm exec jest --runInBand --runTestsByPath ./__tests__/app/job-detail-page.test.tsx` passes, confirming the root suite itself is not the problem
    - `jest.config.js` does not ignore `.worktrees/`
  - Impact:
    - Standard test command becomes nondeterministic and branch-contaminated when developers follow the repo’s worktree-based workflow
  - GitHub issue:
    - #40 `pnpm test가 .worktrees 하위 테스트와 package.json까지 포함해 실패함`
    - https://github.com/cgm-16/vridge/issues/40
