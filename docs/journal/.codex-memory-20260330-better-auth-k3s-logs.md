# 2026-03-30 Better Auth k3s log triage

## User context

- Ori asked how to inspect the real error behind the generic Next.js production Server Components render failure.
- Specific failing path reported: `https://vridge.xyz/api/auth/get-session`.
- Deployment context: production on k3s.

## Repo facts

- Auth route handler lives at `app/api/auth/[...all]/route.ts`.
- Route lazily initializes Better Auth handler via `getHandler()` and `getAuth()`.
- Better Auth config lives at `backend/infrastructure/auth.ts`.
- Required runtime env includes `DATABASE_URL`, `DIRECT_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `NEXT_PUBLIC_APP_URL`, `NEXT_PUBLIC_GA_MEASUREMENT_ID`, `NEXT_PUBLIC_PRIVACY_POLICY_URL`.
- Helm deployment injects `DATABASE_URL`, `DIRECT_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `NEXT_PUBLIC_APP_URL`.
- Current deployment template does not inject `NEXT_PUBLIC_GA_MEASUREMENT_ID` or `NEXT_PUBLIC_PRIVACY_POLICY_URL`.

## Likely debugging angles

- Generic RSC error in production is likely downstream of a server-side throw while fetching or using `/api/auth/get-session`.
- In k3s, primary evidence source is pod/container logs for the Next.js app deployment, then ingress logs if requests are not reaching the pod.
- Missing runtime env is a strong candidate because `backend/config/env.ts` throws on absent/invalid values.
